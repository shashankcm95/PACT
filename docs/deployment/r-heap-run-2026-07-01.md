# R-heap live run — 2026-07-01 (the evidence bundle)

> The first world-anchored HEAP-read hardening in PACT. Spec: `plans/26`; verdict logic:
> `v0/src/identity/heap-read-probe.js`; procedure: `docs/deployment/r-heap-runbook.md`. This record is the
> immutable evidence the hardening claim rests on — the box, the legs, the verdict, and the out-of-band
> attestation, followed by the honest ceiling (an independent honesty-auditor lens graded the claim; its
> findings are folded into §6).

## §0 The honest headline (read first — NS-9 / OQ-NS-6)

This run **HARDENS ONE AXIS**: the signing key cannot be read from the *running broker's memory*, across
**both** the cross-uid boundary (a different unprivileged uid) **and** the same-uid boundary (a distinct
process at the broker's own uid), by **real kernel enforcement** (`ptrace_may_access`) on an unmodified
Linux 6.8 kernel — **non-vacuous** (the L3 positive control proved the key was resident) and **out-of-band
attested**. Per OQ-NS-6 this is world-anchored (real kernel enforcement of a privilege boundary), so
"hardens" is the correct verb, not "narrows."

Scope is **one box, one run, one axis.** It does NOT close: the same-uid **oracle** residual (R2 — an
allowlisted same-uid caller can still trigger a *sign*), the hypervisor / host-admin / root boundary (root
and the VM host are the TCB — L3 itself shows root *can* read the memory), R3 (own-key forgery), or the apex
(external-maintainer-merge). It does NOT protect against code execution *within* the broker process (a
process can always read its own memory). The hardening is **config-conditional** (scope=2 + core/swap lock +
no borrowed-cap binary + broker-as-separate-uid) and is **not** self-enforcing.

## §1 The box

- Host: Apple Silicon (arm64); VM via `multipass` — Ubuntu 24.04.4 LTS, kernel **`6.8.0-124-generic`**.
- LSM list: `lockdown,capability,landlock,yama,apparmor` — **Yama active** (so `ptrace_scope=2` is enforced).
- Lock state at run time: `kernel.yama.ptrace_scope=2`, `kernel.core_pattern=/dev/null` (**apport disabled** —
  it re-set `core_pattern` on boot; see the runbook §2 fix), `swapoff -a`, **zero `cap_sys_ptrace` binaries**
  (snapd's `snap-confine` file-cap stripped + snapd masked), setuid surface reviewed (standard Ubuntu set,
  no ptrace/memory primitive).
- uids: operator `ubuntu`=1000, broker `pactbroker`=999, attacker=1001. Key `/etc/pact/broker.key` mode
  `0600` owned by 999, in a `0755` (traversable) dir.

## §2 L-pre — custody-verify (the file-custody gate; must pass before any heap leg)

Run as the operator (uid 1000): `C0`/`C1`/`C2`/`C3`/`C2.5` all **PASS**, `hostObservableChecksPassed: true`,
`requiresOutOfBandUidConfirmation: true`, and **exit 0** after `--attested-cross-uid`. The load-bearing
`C3-liveness` passed: a real key signed *through* the `sudo -> pactbroker -> wrapper` cross-uid path and
verified as the persona — so a working key genuinely lives behind uid 999, and the host never read it.

## §5 The out-of-band attestation (NS-7 — the human step)

- `id` → uid **1000** (operator, not root).
- `ls -ln /etc/pact/broker.key` → owner **999** (a genuinely different non-root uid).
- `cat /etc/pact/broker.key` (as operator) and as attacker (1001) → **Permission denied**.
- The run log captured `ps -o uid= -p 2534` → **999** (the paused broker ran as pactbroker).

Per `heap-read-probe.js`, the flag records the attestation; it changes the exit code, **not** the proof —
the process↔uid bind is operator-attested, structurally unprovable by the tool.

## §6 Honesty VALIDATE — the ceiling (independent auditor findings folded)

An independent honesty-auditor graded the claim against this evidence. Verdict: the claim is honest within a
tightened ceiling. Folded qualifications:

- **The L3 non-vacuity is genuine** (auditor HIGH-1, refuted): the producing harness
  (`v0/src/identity/heap-read-broker-harness.js`) + `plans/26` are committed (#23) and the harness **pins the
  PEM live** — so the L3 PEM-find is real residency, deliberately in a window *larger* than production's
  load-sign-exit (attacker-favorable, so the result is conservative).
- **L2-core proves core-dump SUPPRESSION, not key-residency-at-crash** (HIGH-2): the throwaway pid is not
  L3-gated; its residency is design-asserted (the harness signs before printing ready), not positive-controlled.
- **The same-uid axis is single-factor** (HIGH-3): it holds solely by `ptrace_scope=2` + no `PR_SET_PTRACER`
  — it has **no uid-separation fallback**, so a scope downgrade or a tracer declaration reopens it, whereas the
  cross-uid axis keeps uid separation as a second wall.
- **First on Linux `ptrace_scope=2`** (MEDIUM-1): the prior three custody signals (R1 / R2-WHO / R2-WHAT) were
  on the user's macOS box; this is the first on the strongest enforcement substrate. The memory axis has a
  code-representativeness dependency R1 lacked — conservative (the harness is attacker-favorable).
- **Attestation is operator-attested, not tool-proven** (MEDIUM-2): the process↔uid bind is the operator's, per
  the tool's own framing.
- **Locks verified at run time, not across a reboot** (LOW-1): `swapoff` was not made reboot-persistent this run.

**Net:** a genuine one-axis hardening (the 4th custody signal, and the first on Linux scope=2), inside a
clearly-bounded ceiling. R2-oracle / hypervisor / R3 / the apex remain OPEN.
